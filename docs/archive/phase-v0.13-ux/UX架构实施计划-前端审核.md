## 计划本身的技术问题（代码已规避，但计划文字需纠正）

1. **P1-1 选区检测方案有 bug（请勿照此实现）**
   计划用 `setInterval` 每 500ms 调用 `selection.$from.node()`。问题：
   - `$from.node()` 返回的是光标**前一个**节点（父块），并非"选中"的节点，对图片等多字符节点会误判；
   - 轮询既浪费又迟钝。
   实际代码已改用 `ctx.get(editorViewCtx)` 在选区变化时直接推导 `selectedType`，更准、更省。**沿用现有方案，不要回退到计划的轮询。**

2. **P1-1 图片分支是唯一还没做的真功能点**
   计划希望「选中图片→宽度滑杆/替换/删除」，当前 `context-bar` 只有 `heading` 和 `text` 两个分支（`selectedType` 类型注释也只有 `null | 'text' | 'heading'`）。这是补起来投入小、体验增益明确的项。

3. **响应式章节是计划最弱的一环**
   计划仅写"深色模式自动继承" + "补充 context-bar/status-bar 的平板字号"，未触及真实风险点。而代码其实已有完整响应式（移动端底部工具栏 + 全屏预览覆盖层 + FAB 浮钮）。计划没覆盖真正的缺口（见下节）。

## 你明确要的「响应式设计」——现状与缺口

**已有（好于计划描述）**
- `<768px`：工具栏转底部固定栏、预览变全屏覆盖层、FAB 切换、状态栏/隐私行精简。
- `768–1024px`：悬浮面板 280px、预览 38vw。
- `≥769px`：桌面三栏 + 面板 push。
- 深色模式：新增组件均用 CSS 变量（`variables.css` + `darkTheme.js`），自动继承 ✅。

**真实缺口（建议补）**
1. **小屏手机（≤375px）工具栏溢出**：12 个图标各 40px ≈ 480px，在 375px 宽会换行/挤压。应改横向滚动或收起次要分组。
2. **无横屏适配**：手机横屏下底部工具栏 + 全屏预览的叠放未处理。
3. **平板预览 38vw 偏窄**：iPad 竖屏 810px 时编辑器与预览都局促，可考虑预览默认收起或可调。
4. **移动端 Onboarding 未单独验证**：全屏覆盖层下文案/按钮在窄屏的观感。
5. 计划承诺的「深色模式自动继承」成立，但新增组件需持续遵守（已遵守）。
