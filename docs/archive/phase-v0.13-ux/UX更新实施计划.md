# 净排 · UX 更新实施计划（v0.14 路线图）

> 角色：用户体验架构师（ArchitectUX）
> 日期：2026-07-21
> 基线：`v0.13.3`（`src/App.vue` ≈ 2800 行单组件）
> 依据：[[UX架构重设计-断点与流畅架构]] 中已定位的 12 处断点（B1–B12）与流畅架构方案
> 硬约束（沿用）：纯前端 · 不登录 · 内容不入库 · 不联网（AI 仅用户主动触发的浏览器直连）；不 reintroduce 本地图片上传（微信过滤 base64 的硬约束不变）

---

## 0. 目标与原则

1. **围绕生命周期重排**：写作 → 排版 → 预览 → 发布，让每个阶段自洽、阶段交界无断崖。
2. **先治根因再治表象**：B10（2800 行单组件）是体验漂移的总根，组件化基座（`tokens.css` + 组件树）先行，后续每个断点修复都落在独立组件上、固化一套 UX 模式。
3. **流程合规**：每项改动先写 **PRD**（功能类）或 **ADR**（架构/交互决策类），复用 `docs/templates/`；完成后更新 `CHANGELOG.md` 与 `RELEASE_CHECKLIST.md`（沿用 v0.13.x 风格，每条附验收清单）。
4. **零回归**：每次合并前 `vite build` 零错误 + `node tests/probe_verify.mjs` 全绿（保护 Enter 幽灵封面、装饰修复）。

## 2. 阶段明细

### M1 — 地基 + 转化闭环（P0 + B10 起步）

**目标**：把根因基座搭起来，并立刻修复最伤转化的发布断层与最易困惑的图片残留、首屏冲突。

| 任务 | 断点 | 建议产物 / 文件 | 验收 | 依赖 |
|------|------|----------------|------|------|
| 1.1 建立设计变量基座 | B10 | 新建 `src/styles/tokens.css`（语义色/间距/圆角/阴影/过渡）；`themes.css`（亮/暗/跟随） | 全站色值改为引用 token，无硬编码 | 无（地基，先做） |
| 1.2 拆分入口协调器 | B9+B10 | 新建 `components/EntryCoordinator.vue`；`App.vue` 仅做编排 | 恢复草稿 > 首次引导 > 空状态 唯一裁决；Onboarding「开始创作」不覆盖现有内容 | 1.1 |
| 1.3 发布向导（闭环） | B4 | 新建 `components/panels/PublishWizard.vue`；`openWeChat` 提升至主发布流 | 复制成功 → 常驻轻引导「打开微信后台粘贴」→ 一键 `window.open(mp.weixin.qq.com)`；附粘贴自检清单（封面/分割线/金句自动生效；外链图正常、base64 需素材库；深色端观感提醒） | 1.1 |
| 1.4 装饰/图片解耦 | B1 | 工具栏标签与 `activePanel` 统一「装饰」；`DecorPanel.vue` 删除「图片」字样与「800×400」文案（方案 A） | 工具栏/面板无「图片」残留，装饰语义单一 | 1.1 |

**M1 验收门槛**：复制→微信 闭环可一步完成；首屏仅一个入口提示；无「图片」残留；`vite build` 零错误 + `probe_verify` 全绿。

### M2 — 一致性收敛（P1）

**目标**：消除「两处都能改外观」「循环切主题」「开关错放」「平板挤压」四类一致性断点。

| 任务 | 断点 | 建议产物 / 文件 | 验收 | 依赖 |
|------|------|----------------|------|------|
| 2.1 统一「外观」面板 | B5 | 合并 `样式` + `设置-品牌` 为 `AppearancePanel.vue`；显式三层叠级联说明（主题→品牌→微调） | 「外观」三层可见可理解；面板内写明「一键应用清空微调但保留品牌」 | M1 基座 |
| 2.2 三态分段控件 | B3 | 新建 `ThemeModeSwitch.vue`（分段 radiogroup + `aria-pressed`），替换 `cycleThemeMode` 循环 | 三态可直达；`aria-pressed` 正确 | 1.1 |
| 2.3 微信深色归位 | B2 | `previewWechatDark` 开关从「发布」组移入「预览」区/预览面板头部，加「(仅预览)」标注 | 预览区内可见、标注清晰，不再混入发布 | 1.1 |
| 2.4 响应式三档重定 | B8 | 明确 ≤768 / 769–1024 / ≥1025 三套；平板面板改「覆盖」非「push」，编辑器最小宽 ≥480px | 三档下编辑器最小宽 ≥480px，面板不挤压 | 1.1 + 组件化推进 |

**M2 验收门槛**：外观三层可理解；三态可直达且 `aria` 正确；微信深色归位并标注；三档断点编辑器不被挤压；回归全绿。

### M3 — 打磨与深化（P2 + B10 收尾）

**目标**：扫清剩余死路/误导，并完成组件化收尾，让架构真正不再漂移。

| 任务 | 断点 | 建议产物 / 文件 | 验收 | 依赖 |
|------|------|----------------|------|------|
| 3.1 上下文栏一致性 | B6 | `EditorArea.vue` 上下文栏：仅在有 ≥1 可用操作时出现；短文给格式化操作（加粗/斜体/链接） | 上下文栏无死路 | M1 基座 |
| 3.2 导入去向明确 | B7 | `ImportPanel.vue`：提纯后确认 + 跳编辑器顶部 + toast「已导入 N 字」 | 导入后有明确去向，不与空状态示例竞争 | M1 基座 |
| 3.3 AI 前置说明 | B11 | `AIPanel.vue` 首次打开提示「需自备 API Key，仅本地直连厂商」 | 首次用 AI 不像「坏了」 | M1 基座 |
| 3.4 复制 toast 条件化 | B12 | 仅当正文含 `data:` URI（base64 本地图）时警告；外链图 `![alt](url)` 不告警 | 正常外链图不再误告警 | M1 基座 |
| 3.5 组件化收尾 | B10 | 按 `UX架构重设计` §3.9 组件树完成拆分：`ToolbarGroup`/`PreviewPanel`/`EditorArea` 等；固化 UX 模式 | `App.vue` 降至编排层（<400 行级），无单组件承载全 UI | M1/M2 各组件 |

**M3 验收门槛**：上下文栏无死路；导入有去向；AI 有前置说明；toast 不再误告警；组件树落地；`vite build` 零错误 + `probe_verify` 全绿 + 全量无障碍基线审计（对比度 ≥ WCAG 2.1 AA）。

## 3. 与既有流程的衔接（PRD / ADR 清单）

按 `docs/STANDARDIZATION.md` 的 6 大流程，本计划各改动对应文档类型：

| 类型 | 对应断点 / 改动 | 说明 |
|------|----------------|------|
| **PRD（功能类）** | B4 发布向导、B9 入口协调器、B7 导入去向、B3 三态控件 | 属新增/重排用户功能，先写 PRD |
| **ADR（架构/交互决策类）** | B1 装饰图片解耦、B5 外观三层模型、B8 响应式三档、B10 组件化拆分 | 记录架构与交互决策，引用既有 `ADR-图片操作管线修复` |
| **组件内调整（随拆分直接修）** | B2 归位、B6 一致性、B11 前置说明、B12 toast 条件化 | 不单独发 PRD/ADR，随对应组件拆分 PR 一并处理 |

- 图片相关（B1/B8/B12）与 `docs/design/ADR-图片操作管线修复.md`、`影响评估-移除图片功能.md` 对齐，**不 reintroduce 本地图片上传**。
- 每个 PR/MR 合并前更新 `CHANGELOG.md` 与 `RELEASE_CHECKLIST.md`，每条附验收清单（沿用 v0.13.x 风格）。

## 4. 执行顺序与依赖图

```
[B10 基座] tokens.css + 组件树
   ├─ 1.1 tokens.css ──┬─ 1.2 EntryCoordinator (B9)
   │                   ├─ 1.3 PublishWizard (B4)   ← 最大价值，尽早
   │                   ├─ 1.4 DecorPanel (B1)
   │                   ├─ 2.1 AppearancePanel (B5)
   │                   ├─ 2.2 ThemeModeSwitch (B3)
   │                   ├─ 2.3 微信深色归位 (B2)
   │                   ├─ 2.4 响应式三档 (B8)
   │                   ├─ 3.1~3.4 打磨项
   │                   └─ 3.5 组件化收尾
   └─ 每拆一组件即固化一套 UX 模式（防漂移）
```

- **先做 1.1**：所有组件拆分都依赖 `tokens.css` 基座，不先搭地基会二次返工。
- **B4 与 B9/B1 相互独立**，可并行开发，但都需等 1.1 完成。
- **M2/M3 大量依赖 M1 组件基座已就绪**，故严格按 M1→M2→M3 推进。

## 5. 风险与回滚

| 风险 | 缓解 / 回滚 |
|------|------------|
| 单组件拆分引入回归（尤其 Enter 幽灵封面、装饰渲染） | 每次拆完跑 `probe_verify.mjs`；保留 `window.__milkdownView` 复现通道；出问题即 revert 该组件 PR |
| 微信深色模拟算法被改动 | v0.13.3 的 `deriveWeChatDarkTheme` 为事实标准，本计划不改算法，仅移动开关位置（B2） |
| 组件拆分范围蔓延 | 严格按 §3.9 组件树边界；不在本计划内新增功能（如本地图片上传） |
| 发布向导误触微信外链策略 | `openWeChat` 仅 `window.open`，不代为操作；自检清单为只读提示，不自动上传 |

## 6. 节奏建议

- 每个里程碑 = 一个版本号（v0.14.0 / .1 / .2），独立 PR、独立回归、独立 CHANGELOG 条目。
- M1 优先排期：发布闭环（B4）是产品价值兑现点，建议作为本计划第一个交付物抢先上线。
- 全部完成后，ArchitectUX 出具「UX 基线达标」验收报告（对照 §5 验收清单与无障碍基线）。
