## 一、本次修复的 4 个 Bug

### Bug #1 — 上下文工具栏调宽度 / 删除 / 替换弹「检测不到该图片」

**现象**：点「替换」「删除」或拖宽度滑杆后，toast 提示「找不到该图片，请重新选中后重试」。

**根因**：`deleteEditingImage` / `replaceEditingImage` 用 `editingImage.value.src`（来自 DOM `img.src`）调用 `matchImageBySrc` 查 markdown 文本。浏览器对 base64 data URL 会做规范化（去掉 padding、改变大小写等），导致 DOM 的 src 与 Markdown 文本中的 src **字符串不完全相等**，模糊匹配也挂了。

**修复**：从 ProseMirror 节点直接读 src（与序列化后的 markdown 一致），而非从 DOM 拿：

```javascript
// 新增辅助函数
const getSelectedImageSrc = () => {
  if (!editingImage.value || !milkdownEditor) return editingImage.value.src;
  try {
    return milkdownEditor.action((ctx) => {
      const view = ctx.get(editorViewCtx);
      const selNode = view.state.selection.node;
      return selNode?.type.name === 'image' ? selNode.attrs.src : editingImage.value.src;
    });
  } catch (_) {
    return editingImage.value.src;
  }
};
```

然后将三个函数中的 `editingImage.value.src` 替换为 `getSelectedImageSrc()`（仅当操作需要匹配 Markdown 中的图片时）。

**影响范围**：`deleteEditingImage`、`replaceEditingImage`、`updateImageWidth`

**风险**：低。fallback 回 DOM src，不会比现在更差。

### Bug #2 — 编辑器与预览界面图片宽度不一致

**现象**：上下文工具栏调宽度到 50%：
- 预览区图片显示 50% ✅
- 编辑器内图片仍是 100% ❌

**根因**：`updateImageWidth` 在 7/18 修复中改为只更新 `imageWidthMap` + `convertMarkdown()`。这样可以正确刷新预览（`buildHtml` 消费 map），但编辑器内的 ProseMirror 图片节点**没有 width 属性**。

**⚠️ 不能回退到 `setNodeMarkup`**：之前在 `updateImageWidth` 中使用 `view.dispatch(tr.setNodeMarkup(...))` 会给图片节点加 width，但 Milkdown 默认 image 序列化器不支持 width → `markdownUpdated` 序列化时丢失 width → 触发 `replaceAll` 重解析 → 节点被重置。这是之前 FIXME B-2 的根本原因。

**修复方案**：直接操作编辑器内 `<img>` 的 DOM style（不走 ProseMirror 事务）：

```javascript
const updateImageWidth = () => {
  if (!editingImage.value || selectedType.value !== 'image') return;
  const src = editingImage.value.src;
  const w = imageWidth.value;
  
  // 1. 更新 imageWidthMap（buildHtml 消费 → 预览正确）
  imageWidthMap.value[src] = w;
  saveImageWidthMap();
  
  // 2. 刷新预览
  convertMarkdown();
  
  // 3. 直接改编辑器 DOM（不经过 ProseMirror，不触发序列化/回写）
  nextTick(() => {
    const pm = document.querySelector('.ProseMirror');
    const imgs = pm?.querySelectorAll('img');
    imgs?.forEach(img => {
      if (img.src === src || img.getAttribute('src') === src) {
        img.style.width = w + '%';
      }
    });
  });
  
  showToast(`图片宽度已设为 ${w}%`, 'success');
};
```

**为什么这是安全的**：
- DOM 修改不触发 ProseMirror 事务 → 不触发 `markdownUpdated` → 不触发序列化 → 不回写到 `markdownText`
- 下次编辑器内容发生变化（用户打字、`replaceAll` 等）时，ProseMirror 重新渲染会清掉这个 DOM style——但此时 `updateImageWidth` 的效果已经被"固化"到下一次渲染中了（宽度的"真相来源"是 `imageWidthMap`，不是 DOM）
- 即使 DOM 修改被重新渲染覆盖，也不会有副作用（图片只是回到 100% 宽度，用户再调一次滑杆就行）

**风险**：低——DOM hack 是临时的视觉效果，不参与任何数据流。

### Bug #3 — 复制到微信公众号后台出现 `...`（六个点）

**现象**：复制 HTML 粘贴到微信编辑器后，图片位置显示 `...`。

**根因**：微信不接受 `data:` 协议的 `<img>` src。base64 内联图片在微信端被过滤并替换为占位符。这不是代码 Bug，是微信平台限制。

**修复**：强化 copy 操作后的 toast 文案，让用户明确知道需要手动上传：

```javascript
showToast(
  'HTML 已复制。⚠️ 图片为本地 base64 数据，微信会替换为「…」——请到微信素材库上传原图后替换。',
  'warning',  // 新增 warning 级别提示
  6000        // 延长显示时间
);
```

`showToast` 当前只支持 `success` 和 `error` 两种类型。需要扩展第三种 `warning` 类型（橙色背景）。

**影响范围**：`copyHtml` 函数中的 toast，`showToast` 函数签名。

**风险**：无。纯文案改动。

### Bug #4 — 删除图片后出现 `---` 分割线残留

**现象**：撤销图片插入后，编辑器出现 `---` 文字，无法用 Enter 调整，无法删除。

**状态**：🟡 **先调查，不立即修。** 根因尚未确认。几种可能：
1. `insertImage` 间接触发了装饰块逻辑（但 7/18 已确认 `insertImage` 不走 `insertDecorBlock`）
2. 图片被 Milkdown 解析为与 `---` 水平线节点相邻的事务
3. `replaceAll` 重解析时 Markdown 文本意外包含了 `---`

**调试步骤**：
- 插入图片 → 打开 Console → 检查 `markdownText.value` 的原始内容
- Ctrl+Z → 再次检查 `markdownText.value`
- 对比前后差异，定位 `---` 是否在 Markdown 文本中
- 如果在 → 追踪它何时被插入
- 如果不在但编辑器显示了 → 是 ProseMirror 节点问题

**调完根因后再决定方案。** 不要猜测性修。
