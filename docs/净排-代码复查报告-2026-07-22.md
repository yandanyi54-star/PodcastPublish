## 二、本次修复的问题

### 🔴 真实缺陷：ESLint 从未审查过 `.vue` 文件
`eslint.config.js` 只挂了 `eslint-plugin-vue` 插件，却**没配 `vue-eslint-parser`**，导致所有组件文件（App.vue 及 6 个组件）都被默认识别成普通 JS，直接报 `Unexpected token <` 解析失败 → 长期"假绿"，Vue 代码无人审查。
**修复**：`.js` 用 espree、`.vue` 用 `vue-eslint-parser`（`<script>` 块交 espree）；组件命名规则放宽为 `PascalCase`（匹配现有代码风格）。

### 🟡 代码整洁度（不影响运行，已清理）
1. `App.vue`：删除未使用的 `THEME_NAMES` 导入与 `getDefaultStyle` 解构、删除死代码 `const cmd`。
2. `App.vue`：`let selTimer` → `const selTimer`（从不重赋值）。
3. `buildHtml.js`：图片宽度正则去掉未使用的 `after` 捕获组。
4. `forceParagraphEnter.js`：删除未使用的 `NodeSelection` 导入。
5. `useBrand.js`：`catch (_)` → `catch {}`（ES2019 可选捕获，更干净）。
6. `App.vue`：6 处 DEV 调试 `console.log` → `console.warn`（均被 `import.meta.env.DEV` 包裹，生产构建摇树移除，不影响线上）。

## 三、重点排查的「疑似 bug」（已排除，切勿误改）

**1. `editor.action(fn)` 不自动 dispatch，插入/替换会不会失效？**
不会。`replaceAll`/`replaceRange` 在内部**直接 `view.dispatch`**（`replace-all.ts`/`replace-range.ts` 源码已确认）。所以 `setEditorMarkdown`、选区替换、插入装饰块都真实改写编辑器。✅ 不是 bug。

**2. DOMPurify 会不会把主题的内联 `style` 净化掉？**
不会。用 jsdom 实测：DOMPurify 3.1.5 默认**保留内联 `style` 与 `data-decor` 属性**。这是"主题→HTML"整条链路的安全前提，已验证。✅ 不是 bug。

**3. 两条 `watch(markdownText)`（异步重渲染 + 同步推编辑器）会不会死循环 / 光标错位？**
不会。配 `isFromEditor`（编辑器→文本）与 `isSettingEditor`（程序化改写）双闸门，审阅逻辑与历史"Enter 幽灵封面"修复一致，无自触发回路。✅ 不是 bug。

用户确认后，处理第四节中两项体验级小问题：

### ✅ 1. 全局快捷键在表单控件聚焦时不再误触发（`src/App.vue` `onKeydown`）
在原有 `e.isComposing`（IME 保护）之后新增表单控件判断：
```js
// 表单控件（输入框/文本域/下拉）聚焦时不触发全局快捷键，避免干扰 AI 提示词、品牌色、选项填写
const tag = e.target?.tagName;
if (tag === 'INPUT' || tag === 'TEXTAREA' || tag === 'SELECT') return;
```
**效果**：焦点在 AI 提示词框 / API Key 输入框 / 品牌色输入框 / 下拉选项时，`Ctrl+Shift+C`、`Ctrl+Shift+P`、`Ctrl+S`、`Ctrl+/` 都不再触发应用动作（避免打断填写）。编辑器本身（contenteditable 的 `.milkdown` div，tag 为 `DIV`）不受影响，`Ctrl+S` 保存等仍可用。
